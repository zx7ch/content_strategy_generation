## 2. Architecture Context

### System Position
```text
Workbench creates task and opens Xiaohongshu
  -> Server owns active task, token issuance, capture persistence, snapshot version
  -> Background syncs active task and orchestrates capture requests
  -> Content script extracts visible page data and renders page-level capture UI
  -> Popup displays connection/task/page status and fallback controls
```

### Tech Stack
- Language/runtime: Python FastAPI server, vanilla JS web app, Chrome MV3 extension.
- Primary services: standalone experiment server under `experiments/xhs_extension_mvp/server`.
- Execution pattern: local browser-extension workflow using background/content-script messaging and local HTTP API calls.
- Key behavioral constraints: no normal-path manual token transfer, no production backend coupling, visible content only, single active task for MVP.

### Constraints
- Keep the MVP isolated from formal `app/api/routes/router.py`, production `Session/Job/Worker/LangGraph`, and `data/xhs_agent.db`.
- Do not introduce implementation code in Phase 0.
- Keep token as runtime credential, not user-facing workflow state.

### 3.2 Interface Design

**Frozen Endpoints:**
```text
GET  /api/extension/health
GET  /api/extension/active-task
POST /api/tasks/{task_id}/activate
POST /api/tasks/{task_id}/active-search-context
GET  /api/tasks/{task_id}/snapshot
GET  /api/tasks/{task_id}/candidate-directions
POST /api/extension/capture
```

**Primary Data Structures:**
```text
ActiveTask:
  task_id, capture_token, topic, created_at, activated_at, status,
  snapshot_version, capture_count, candidate_count

TaskSnapshot:
  task_id, snapshot_version, updated_at, candidate_count, capture_count,
  last_capture_result

ActiveSearchContext:
  task_id, query, source, opened_at

CaptureRequest:
  task_id, request_id, tab_id, page_url, page_type, query_text, visible_items

CaptureResult:
  task_id, request_id, ingestion_run_id, snapshot_version, captured_count,
  new_count, duplicate_count, status, error_summary
```

### 3.3 Algorithm & Logic Flow

**Core Flow:**
```text
Workbench creates and activates task
  -> content script initializes on supported XHS page
  -> content asks background for active task
  -> background checks health and active task
  -> content renders panel when active task exists
  -> user captures visible page
  -> background submits capture with runtime token
  -> server merges and increments snapshot_version
  -> page shows capture result
  -> workbench polling observes snapshot_version change and refreshes candidates
```
