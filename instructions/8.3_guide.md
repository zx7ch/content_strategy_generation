## 1. Task Context

### Scope Boundary
- **Task ID**: 8.3
- **Task Name**: Phase 2: Popup Downgrade to Status Panel
- **Phase**: XHS Extension MVP UX Optimization / Phase 2
- **Dependencies**: 8.2 active task runtime foundation is implemented.
- **Task Goal**: Make popup a lightweight runtime status panel instead of a business-state transfer surface or capture product surface.

### In Scope
- Move Server URL into an advanced/settings area.
- Show connection state, active task, current page status, and visible note count.
- Keep resync and open workbench controls.
- Improve user-facing fallback messages for server-down, no-task, unsupported-page, and content-script-missing states.
- Preserve no-token normal flow and avoid any visible capture token input.

### Out Of Scope
- Page-level floating capture panel.
- Page-level capture panel implementation.
- Workbench auto-refresh.
- Multi-task switching, SSE, automatic scrolling, page turning, or simulated clicking.

### Required Deliverables
- Production: `extension/src/popup.html`, `extension/src/popup.js`, and any small background message support needed for page status.
- Tests: update `tests/unit/test_xhs_extension_mvp.py` static extension contract checks.
- Spec/Docs: append Phase 2 implementation record to `spec.md`.

### Acceptance Criteria
- [ ] AC1 Popup primary copy and layout present as a status panel, not "XHS MVP Capture".
- [ ] AC2 Server URL is placed in an advanced/settings disclosure, not top-level primary UI.
- [ ] AC3 Popup displays connection state, active task, page status, and visible note count.
- [ ] AC4 Popup includes `Open workbench` and `Resync active task` controls, but no capture control.
- [ ] AC5 No visible token input or token wording appears in popup HTML.
- [ ] AC6 Fallback messages are user-oriented for server down, no active task, unsupported page, and content script recovery.

### Residual Obligations
- **Relevant OPEN Residuals**: none in a formal tracker for this standalone experiment.
- **Current-Phase Carry-Forward Items To Re-check**: Phase 3 must move the main capture action into the Xiaohongshu page floating panel.
- **Resolved By This Task**: Popup no longer owns business-state transfer and becomes status UI.
- **Deferred / Blocked**: Page capture panel remains Phase 3 work.

### Contract Inventory
- Upstream contracts: background messages `HEALTH_CHECK`, `ACTIVE_TASK_RESYNC`, `PAGE_STATUS_REQUEST`, `CAPTURE_VISIBLE_PAGE`, `CONTENT_SCRIPT_ENSURE`.
- Downstream contracts: Phase 3 content panel will become primary capture path.
- Files/interfaces with compatibility risk: popup DOM ids and static tests.

### Test Requirements
- **Test File**: `tests/unit/test_xhs_extension_mvp.py`
- **Layer**: unit/static contract check.
- **Test Scenarios**:
  1. Popup title/copy describes a status panel.
  2. Server URL lives in advanced settings.
  3. Token input/wording is absent from popup HTML.
  4. Page status and visible count ids exist.
  5. Workbench/resync controls exist and capture controls are absent.
  6. Popup JS uses `PAGE_STATUS_REQUEST`, `ACTIVE_TASK_RESYNC`, and product fallback messages.
