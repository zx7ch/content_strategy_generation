## 1. Task Context

### Scope Boundary
- **Task ID**: 8.5
- **Task Name**: Phase 4: Auto-Refresh Workbench Snapshot
- **Phase**: XHS Extension MVP UX Optimization / Phase 4
- **Dependencies**: 8.2 server active task/capture foundation and 8.4 page capture panel are complete.
- **Task Goal**: Make the workbench automatically detect capture updates and refresh candidate directions without requiring manual snapshot refresh.

### In Scope
- Add task snapshot version fields to server responses.
- Add `GET /api/tasks/{task_id}/snapshot` for lightweight polling.
- Add `GET /api/tasks/{task_id}/candidate-directions` for version-change refresh.
- Start 2000ms workbench polling after task creation/snapshot load.
- Refresh workbench modules when `snapshot_version` changes.
- Show current task, capture count, candidate count, and recent update time in the workbench.

### Out Of Scope
- SSE/live push.
- Browser notification/toast system beyond status copy.
- Changing capture semantics.
- Manual snapshot refresh.

### Acceptance Criteria
- [ ] AC1 Server task snapshot includes `snapshot_version`, `updated_at`, `candidate_count`, and `capture_count`.
- [ ] AC2 `GET /api/tasks/{task_id}/snapshot` returns lightweight version metadata.
- [ ] AC3 `GET /api/tasks/{task_id}/candidate-directions` returns refreshed workbench data for version changes.
- [ ] AC4 Workbench polls every `2000ms` while visible and a task exists.
- [ ] AC5 Version changes refresh candidates/recommendations/collection summary automatically.
- [ ] AC6 Workbench top status shows current task, capture count, candidate count, and recent update time.

### Workbench Flow
```text
task loaded
  -> store current snapshot_version
  -> start 2000ms polling
  -> if lightweight version changes
      -> fetch candidate directions/full workbench snapshot
      -> render task/query/collection/recommendation modules
      -> show auto-refresh status
```
