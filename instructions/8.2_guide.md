## 1. Task Context

### Scope Boundary
- **Task ID**: 8.2
- **Task Name**: Phase 1: Active Task Runtime Foundation
- **Phase**: XHS Extension MVP UX Optimization / Phase 1
- **Dependencies**: 8.1 Phase 0 interface record is complete.
- **Task Goal**: Remove manual token transfer from the normal capture path by making the server own the active task and the extension background own runtime synchronization.

### In Scope
- Add server-side active task state.
- Mark newly created tasks active and support explicit task activation.
- Add active task, active search context, extension health, and extension capture endpoints.
- Add Chrome MV3 background runtime state with active task sync.
- Let popup display connection state and active task, and submit capture through background without requiring token input.
- Let content script request active task on initialization.

### Out Of Scope
- Page-level floating capture panel.
- Workbench snapshot polling and candidate auto-refresh.
- Full popup redesign beyond runtime status and non-token normal capture.
- Multi-task switching, SSE, auto scrolling, page turning, or simulated clicking.

### Required Deliverables
- Production: server models/storage/app endpoints, extension manifest/background/popup/content runtime changes.
- Tests: unit/API coverage for active task and extension capture; lightweight static checks for extension runtime wiring.
- Spec/Docs: this guide only; no spec wording changes required unless implementation discovers a contract gap.

### Acceptance Criteria
- [ ] AC1 Creating a task marks it as the single active task and `GET /api/extension/active-task` returns its topic, token, timestamps, status, snapshot version, capture count, and candidate count.
- [ ] AC2 `POST /api/tasks/{task_id}/activate` is idempotent and switches the active task; missing task returns 404.
- [ ] AC3 `POST /api/tasks/{task_id}/active-search-context` stores query/source/opened_at and returns active task context.
- [ ] AC4 `GET /api/extension/health` returns connection health and whether active task exists.
- [ ] AC5 Popup can sync active task and submit capture via background using runtime token, without requiring a user-pasted token.
- [ ] AC6 Content script initializes by requesting active task from background.

## 2. Architecture Context

### System Position
```text
Workbench creates task
  -> Server persists task and active task pointer
  -> Background syncs active task and stores runtime state
  -> Popup requests runtime status and capture orchestration from background
  -> Content script requests active task during initialization
```

### Tech Stack
- Python FastAPI + SQLite for the standalone MVP server.
- Chrome MV3 service worker for background runtime.
- Vanilla JS popup/content scripts.
- Pytest + FastAPI TestClient for task-scoped verification.

### Constraints
- Keep legacy `/mvp/*` endpoints working for current workbench compatibility.
- Do not require users to paste capture token in the normal popup capture path.
- Store runtime token in background/chrome storage only as implementation state.
- Preserve visible-content-only capture behavior.

```text
HEALTH_CHECK

```

### 3.3 Algorithm & Logic Flow

**Create/Sync Flow:**
```text
POST /mvp/tasks
  -> create task
  -> create capture token
  -> set active task
  -> extension GET /api/extension/active-task can now return runtime context
```

**Popup Capture Flow:**
```text
popup CAPTURE_VISIBLE_PAGE
  -> background syncs active task
  -> background ensures/extracts content script payload
  -> background POST /api/extension/capture with X-Capture-Token
  -> popup renders imported/candidate counts
```

### 3.4 Implementation Checklist
- [ ] Add SQLite tables for active task and active search context with migration-safe creation.
- [ ] Add storage helpers for active task read/write, activation, search context, and active-task response assembly.
- [ ] Add Pydantic models for active task, health, active search context, and extension capture.
- [ ] Add server routes and keep existing `/mvp/*` behavior backward compatible.
- [ ] Add `extension/src/background.js` and register it in manifest.
- [ ] Update popup to call background for health/sync/capture and not require token input for normal capture.
- [ ] Update content script to request active task on initialization while preserving existing extraction.
- [ ] Add tests for AC1-AC6.

### 3.5 Error Handling Strategy
- No active task returns `200` with `active_task: null` and `error_summary.code = no_active_task`.
- Missing task activation/search context returns `404`.
- Invalid extension capture token returns `401`.
- Background maps unreachable server to `server_unavailable`.
- Content script missing triggers background injection retry.
